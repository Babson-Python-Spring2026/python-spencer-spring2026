Using AI While Staying in Control of the Code

A cleaned-up guide based on a real iterative coding conversation. The point is not to let AI write everything. The point is to use AI as a tool while the human remains responsible for the design, the rules, the invariants, the naming, the file structure, and the final judgment about what is correct.

Central idea

AI should not be the source of truth. The source of truth should be your files, your rules, your data model, and your reasoning.

The human stays in control by specifying constraints, checking outputs, catching mistakes, and forcing the AI to justify or revise code.

1. What this conversation demonstrated

The conversation was not a case of saying, “Write the whole system,” and blindly accepting the answer. Instead, it showed a much healthier workflow:

  • Start with a narrow function.
  • Clarify behavior before coding.
  • Refuse hidden assumptions.
  • Test against actual files.
  • Catch wrong reasoning quickly.
  • Refactor names and architecture as understanding improves.
  • Build the system in layers.
Good pattern

Ask AI for one function, one rule, or one transformation at a time. That forces both the AI and the human to stay close to the state of the system.

2. The control pattern: human first, AI second

Human controls AI helps with
File structure, naming, requirements, invariants, edge cases, correctness checks Drafting code, revising code, explaining syntax, spotting inconsistencies, cleaning formatting
Deciding what the program should do Translating those decisions into Python
Deciding when an answer is wrong Trying again under tighter constraints
Key principle: AI is not the designer. AI is a coding assistant working inside a design the human defines.

3. A better way to prompt AI for code

The strongest parts of the conversation came from prompts like these:

  • “Do not make assumptions.”
  • “Do you have everything you need?”
  • “Restate the goal first.”
  • “What is still ambiguous?”
  • “This is not correct. Redo.”
Weak prompt

Write the whole portfolio system.

This invites guessing, overengineering, and hidden assumptions.

Strong prompt

Write only the function that lists transactions for a ticker. It should prompt until ticker input is valid, sort by date, print aligned columns, blank line every 5, pause every 15, and not clear the screen.

This keeps the AI inside a well-defined box.

4. The most important habit: force assumptions into the open

One of the strongest habits in the conversation was repeatedly asking whether the AI really had enough information to write correct code.

Danger sign

If the AI starts inventing behavior that you never specified, the problem is usually not just the answer. The problem is that the design contract was still incomplete.

Examples of missing details that had to be pinned down before code could be trusted:

  • Should a function print or return a value?
  • Should output be sorted?
  • What counts as a valid date?
  • How should splits affect shares?
  • How should splits affect average cost?
  • What happens when a position crosses through zero?
  • Should positions exist for all market dates or only transaction dates?

5. Build the system in layers, not all at once

The conversation gradually moved toward a very strong architecture. Instead of trying to jump straight to “portfolio by date,” it decomposed the problem into intermediate artifacts.

transactions.json
   ↓
stocks_by_date.json
   ↓
cash_by_date.json
   ↓
portfolio_<date>.json

That is a much better teaching pattern than trying to compute everything in one giant function.

Layer Purpose
stocks_by_date.json Reconstruct stock positions by market date, including split adjustments
cash_by_date.json Reconstruct cash by market date using transactions, dividends, and prior stock state
portfolio_<date>.json Combine stocks and cash for a requested date
Why this matters pedagogically

Students learn that large systems should be built from intermediate states and reusable components. That is computational thinking, not just syntax production.

6. STI stayed in charge of the design

The strongest conceptual thread in the conversation was the repeated use of STI: State, Transitions, Invariants.

Example: stock positions by date

  • State: for each ticker, shares and average cost
  • Transitions: buys, sells, splits
  • Invariant: at the end of each market date, the stored position reflects all prior transitions through that date

Example: cash by date

  • State: one number, cash
  • Transitions: contributions, withdrawals, buys, sells, dividends
  • Invariant: cash on date d reflects all cash flows through that market date

Using AI this way is much better than asking for code without a model. The model comes first. The code comes second.

7. Human corrections made the design better

The AI produced several drafts that were incomplete or wrong. That was not a failure of the process. It was part of the process. The important thing is that the human noticed the problems and tightened the rules.

Examples of corrections that improved the system:

  • Removing transaction-entry code from a program that should only list transactions
  • Changing a self-import that caused a Python import error
  • Fixing the average-cost split rule so that total cost basis stays invariant
  • Requiring positions for every market date, not just transaction dates
  • Renaming positions_by_date.json to stocks_by_date.json for clarity
Important lesson

AI can draft quickly, but naming, decomposition, and correctness still require deliberate human judgment.

8. Good reasons to slow AI down

In the conversation, progress improved whenever the human forced the AI to go slower.

  • Ask for one function, not the whole program.
  • Ask whether anything is still unspecified.
  • Ask the AI to restate the goal.
  • Check the file paths.
  • Check the exact output format.
  • Verify with real examples from the dataset.
Bad workflow:
AI writes whole system → human glances at it → hidden errors survive

Better workflow:
Human defines one piece → AI drafts → human tests against files → human revises rules → AI updates

9. Good prompts for staying in control

These are excellent control prompts for students and instructors:

Restate the goal first.
Do you have everything you need?
What are you assuming?
Do not make assumptions.
What rule is still ambiguous?
Write only the code for this function.
Do not add any other functionality.
Why is this answer correct?
What exact lines need to change?

10. Why this is not “just letting AI do the work”

If students simply paste a broad prompt and accept the result, that is weak learning. But the workflow shown here is different.

  • It requires reading code.
  • It requires spotting errors.
  • It requires designing invariants.
  • It requires understanding edge cases.
  • It requires deciding what should happen before asking AI to write it.
Better interpretation

AI is being used as a junior programmer whose work must be supervised, corrected, and constrained.

11. A suggested classroom rule set

  1. Define the state before asking for code.
  2. Define the transitions before asking for code.
  3. State at least one invariant before asking for code.
  4. Ask for one function at a time.
  5. Ask the AI whether anything is still ambiguous.
  6. Check the answer against real data, not just plausibility.
  7. Revise names and file structure when needed.
  8. Treat generated code as a draft, not as truth.

12. What students should learn from this

The point is not merely to get code that runs. The deeper lesson is that a good programmer:

  • keeps control of the design,
  • thinks in terms of state and transitions,
  • builds systems in layers,
  • tests against real data,
  • and treats every answer, even an AI answer, as something to inspect.
Bottom line

The best use of AI in programming is not surrender. It is disciplined collaboration under human control.
