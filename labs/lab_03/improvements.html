Improving the Portfolio Functions

A cleaned-up guide to improving the six major functions in the portfolio system. The goal is not merely to make the code run. The goal is to make each function clearer, more testable, easier to debug, and more aligned with STI: state, transitions, and invariants.

Main idea

Improve each function by making it do one job clearly, moving validation and formatting into helpers, and making the system logic explicit.

1. create_transaction(...)

This function should be the strict gatekeeper for event quality.

Improvements

  • Validate date against mkt_dates.json.
  • Validate ticker against ticker_universe.json.
  • Validate price against prices_dates.json using the ±15% rule.
  • Separate stock input from cash input more explicitly.
  • Move record_number generation into a helper.
  • Sort only once after append.
  • Use specific exceptions instead of generic except.

Suggested helpers

get_valid_date(...)
get_valid_transaction_type(...)
get_valid_ticker(...)
get_valid_price(...)
next_record_number(...)

This makes transaction creation a coordinator instead of a giant mixed-purpose function.

2. list_transactions_for_ticker(...)

This function works, but it mixes several jobs together.

Improvements

  • Separate data retrieval from printing.
  • Create one helper for formatting shares.
  • Create one helper for printing the header.
  • Keep validation separate from listing.
  • Allow both interactive and non-interactive use if desired.

Suggested decomposition

get_valid_ticker(...)
get_transactions_for_ticker(ticker)
print_transactions_table(transactions)

This makes the logic easier to test and reuse.

3. build_stocks_by_date(...)

This is one of the most important functions in the whole system.

Improvements

  • Rename internal pieces more explicitly.
  • Separate split logic from trade logic.
  • Separate “update one position” from “snapshot full stock state.”
  • Document crossing-zero rules clearly.
  • Normalize numeric types consistently.
  • Add a verification helper for tracing one ticker across dates.

Suggested structure

group_transactions_by_date(...)
build_split_lookup(...)
apply_split(position, split_record)
apply_transaction(position, transaction)
make_snapshot(positions)

Important comment block

# For each market date:
# 1. carry forward prior positions
# 2. apply splits for that date
# 3. apply stock transactions for that date
# 4. store snapshot

This makes the transition order explicit and easier to trust.

4. build_cash_by_date(...)

This is another central function because cash depends on several different sources.

Improvements

  • Separate transaction cash flows from dividend cash flows.
  • Document the dividend ordering rule clearly.
  • Define one helper for daily transaction cash flow.
  • Define one helper for daily dividend cash flow.
  • Round only at storage or output, not at every intermediate step.
  • Process every market date, not just transaction dates.

Suggested structure

group_transactions_by_date(...)
build_position_lookup_for_date(...)
get_transaction_cash_flow_for_date(...)
get_dividend_cash_flow_for_date(...)

Then the main daily logic becomes very simple:

cash = prior_cash
cash += transaction_cash_flow
cash += dividend_cash_flow

5. get_cash_by_date(...)

This should be a small user-facing reporting function.

Improvements

  • Rebuild cash fresh every time.
  • Validate date through one shared helper.
  • Format output with commas and two decimal places.
  • Keep the printout clean and minimal.
  • Avoid duplicating file-loading logic if helpers already exist.

Ideal shape

date = get_valid_date(...)
build_cash_by_date()
cash_by_date = load_cash_by_date()
print_cash(date, cash_by_date[date])

This function should not contain business logic. It should validate, rebuild, read, and display.

6. get_portfolio_by_date(...)

This is the final composition layer. It should feel like assembly, not invention.

Improvements

  • Separate portfolio construction from portfolio printing.
  • Separate market-price lookup from display formatting.
  • Define one helper for cash position creation.
  • Define one helper for stock position enrichment.
  • Use one common formatter for integer shares versus cash.
  • Sort output predictably.
  • Write JSON with a consistent field order.

Suggested structure

get_valid_date(...)
build_stocks_by_date()
build_cash_by_date()
build_price_lookup_for_date(...)
make_stock_portfolio_rows(...)
make_cash_portfolio_row(...)
print_portfolio_table(...)
save_portfolio_for_date(...)

Cross-Function Improvements

A. Centralize file loading

Use common helpers for reading and writing JSON.

load_json(path)
save_json(path, data)

B. Centralize path definitions

Keep all key paths in one place so they do not drift across files.

BASE_FOLDER
TRANSACTIONS_FILE
PRICES_DATES_FILE
MKT_DATES_FILE
...

C. Centralize validation helpers

Especially for:

  • valid date
  • valid ticker
  • valid transaction type
  • valid price

D. Separate computation from display

A strong rule:

one function computes data
another function prints it

This improves testability and debugging.

E. Add verification helpers

Examples:

print_ticker_path("AAPL", start_date, end_date)
check_portfolio_date(date)

These are useful for debugging and teaching.

F. Use docstrings that define STI

For each function, specify:

  • state it reads
  • transitions it applies
  • invariant it preserves
def build_cash_by_date():
    """
    State read:
        transactions, prices, stocks_by_date

    Transition logic:
        applies daily transaction cash flows and dividends

    Invariant:
        cash_by_date[d] equals cumulative cash through date d
    """

Suggested Priority Order

Priority Function Reason
1 build_stocks_by_date Backbone of splits, positions, and later valuation
2 build_cash_by_date Combines transactions, dividends, and stock state
3 get_portfolio_by_date Final composition layer for the full portfolio snapshot
4 create_transaction Important gatekeeper for input quality
5 list_transactions_for_ticker Useful reporting function, but less structurally central
6 get_cash_by_date Small reporting wrapper once the builder is solid
Bottom line

Improve each function by making it do one job clearly, moving validation and formatting into helpers, and making the STI logic explicit in the code.

This document summarizes recommended improvements for the six major functions in the portfolio system and highlights cross-function design improvements that make the whole system cleaner, easier to test, and easier to teach.
