Computational Thinking Reference

Input, State, Transitions, Invariants, State Space, and Solution

Why These Concepts Matter

Computational thinking means more than writing code. It means understanding a problem as a system: what information is given, what information changes, how it changes, what must remain true, and what counts as a successful result.

Main idea: A correct program starts with valid inputs, represents the changing state clearly, applies valid transitions, preserves invariants, and reaches an acceptable final state.

In this exam, you may use AI, but you are responsible for the computational thinking. AI can help with implementation, syntax, debugging, and explanation. You must still be able to explain your design choices.

Running Example: Tic Tac Toe

We will use Tic Tac Toe to explain the main ideas.

A Tic Tac Toe board might look like this:

X | O | X --------- | X | --------- O | |

The game begins with an empty board. Players alternate turns. X usually goes first. A game ends when X wins, O wins, or the board is full and no player has won.

1. Inputs

Inputs are the information given to the problem that does not change while the program is solving that problem.

Inputs define the starting conditions, rules, or constraints. They are not the same as state, because inputs usually do not evolve step by step during the computation.

Tic Tac Toe Example

Possible inputs include:

  • board size: 3 by 3
  • players: X and O
  • rule that X moves first
  • rule that players alternate turns
  • rule that a player wins with three in a row

Separating inputs from state is important because it clarifies what is fixed and what changes.

2. State

State is the minimum amount of changing information needed to describe the current situation of the problem.

State should be sufficient but not unnecessarily large. If the program has the current state, it should have enough information to decide what can happen next.

Tic Tac Toe Example

The state of a Tic Tac Toe game includes:

  • the current board
  • whose turn it is

For example:

X | O | X --------- | X | --------- O | |

This board alone may not be enough unless we can determine whose turn it is. Usually we can derive that from the number of Xs and Os, but it still belongs in our thinking about state because it determines the next move.

In your exam answers, explain what information changes over time. That changing information is your state.

3. Why Separate Inputs from State?

Separating inputs from state gives us a clearer focus on transitions.

Concept Question It Answers Tic Tac Toe Example
Inputs What is given and fixed? 3 by 3 board, X goes first, three in a row wins
State What changes as the problem unfolds? Current board and current player
Transitions How is the state allowed to change? A player places X or O in an empty square

Once inputs are separated from state, transitions become easier to identify: transitions are the valid ways the changing information can change.

4. Transitions

Transitions are the valid operations that change the system from one state to another state.

A transition takes a current state and produces a new state. In code, a transition often becomes a function, a loop step, a recursive call, or an update to a data structure.

Tic Tac Toe Example

A transition is one legal move:

  • X places an X in an empty square
  • O places an O in an empty square

Example transition:

Before:

X | O | X --------- | X | --------- O | |

O moves in the bottom middle square:

After:

X | O | X --------- | X | --------- O | O |

An invalid transition would be placing a mark in an occupied square, allowing the same player to move twice, or continuing the game after someone has already won.

5. Invariants

Invariants are conditions that must always be true for every valid state of the system.

Invariants define what makes a state legal or valid. They also help us check whether our transitions are correct. A transition should never produce an invalid state.

Tic Tac Toe Example

Some Tic Tac Toe invariants are:

  • Each square is either empty, X, or O.
  • X and O cannot occupy the same square.
  • The number of Xs is either equal to the number of Os or exactly one more than the number of Os.
  • If X goes first, O can never have more moves than X.
  • No moves are allowed after the game has ended.

Invariants are not just comments. They are design rules. Good code should be organized so these rules are preserved as the state changes.

6. State Space

The state space is the set of all possible valid states that can be reached from the initial state by valid transitions.

The state space is not just any possible arrangement of symbols. It includes only states that obey the rules and can actually occur through legal transitions.

Tic Tac Toe Example

The Tic Tac Toe state space includes all valid boards that can occur during a legal game:

  • the empty board
  • boards after one move
  • boards after two moves
  • boards where X wins
  • boards where O wins
  • full boards that end in a tie

However, not every board with Xs and Os is part of the valid state space.

Invalid board example:

O | O | O --------- O | X | --------- X | |

This is invalid if X goes first, because O has too many moves. It violates the invariant about the number of Xs and Os.

7. Terminal States

A terminal state is a valid state where the process stops.

Terminal states are final states. They are not the same as solutions.

Tic Tac Toe Example

The terminal states are:

  • X wins
  • O wins
  • the game is a tie

Example terminal state where X wins:

X | X | X --------- O | O | --------- | |

This board is an end state. It tells us the result, but not how the game got there.

8. Solution

A solution is an ordered sequence of valid transitions that moves the system from the initial state to a terminal state.

Equivalently, a solution can be viewed as a path through the state space.

A solution is not just the final answer. “X wins” is an outcome. The solution is the sequence of moves that led to that outcome.

Tic Tac Toe Example

One solution path might be:

Start:
  |   |  
---------
  |   |  
---------
  |   |  

1. X moves top left
2. O moves middle left
3. X moves top middle
4. O moves middle middle
5. X moves top right

End:
X | X | X
---------
O | O |  
---------
  |   |

The terminal state is “X wins.” The solution is the ordered sequence of moves that produced that result.

Common Mistakes to Avoid

These mistakes usually lead to weak designs or confusing code:

  • Confusing the state space with a solution.
  • Describing only the final state instead of the sequence of transitions that gets there.
  • Including fixed inputs as if they were changing state.
  • Defining state that is not minimal, such as storing extra values that can be derived when needed.
  • Forgetting that transitions must preserve invariants.
  • Writing code first and only explaining the state afterward.

Tic Tac Toe Examples

Weak Statement Better Statement
“The solution is X wins.” “X wins is a terminal state. A solution is the ordered sequence of moves that leads to X winning.”
“The state is the rules of Tic Tac Toe.” “The rules are inputs. The state is the current board and whose turn it is.”
“Any board with Xs and Os is in the state space.” “Only valid, reachable boards are in the state space.”

Check Yourself Before Coding

Before you ask AI for code, answer these questions yourself:

  1. What information is fixed input?
  2. What information changes over time?
  3. What is the minimal state needed to continue solving the problem?
  4. What transitions are allowed to change the state?
  5. What conditions must always remain true?
  6. What are the terminal states?
  7. What does a solution path look like?

Your AI transcript should show that you controlled these design decisions. AI may help you implement the code, but you are responsible for defining the problem structure.
