Minimum market concepts and price ideas

Only the concepts needed for this system.

The minimum concepts you need

ConceptMinimal system meaning
StockA thing identified by a ticker that can have a price and a number of shares.
DividendA cash event. If you own the stock, cash is distributed to you.
SplitA share transformation. Share count changes and price scale changes, but value should not jump just because of the split.
Spin-offA new asset appears. In this teaching dataset, selected spin-offs are simplified and treated like dividend-style cash events.
Raw priceThe actual day price used for valuation in the rebuilt file.
Adjusted priceA helper series. In the rebuilt file, it is adjusted only for true splits.
Raw price is the primary input for valuation. Adjusted price is a helper series.

A guiding principle — follow the money

In finance, a useful rule is: follow the money. In this portfolio system, you want to represent history as closely as possible. For that reason, you want to use the actual price that occurred, also called the raw price.

Vendor-style data from sources like yfinance is mainly designed for analyzing returns of individual stocks, not for valuing portfolios through time. That is why the teaching files were rebuilt to keep more of the important events explicit.

In this system Price → what the market did Cash → what was paid out or added Shares → how ownership changed

Short example — why a split is different

Before a 2-for-1 split Shares = 10 Raw Price = 100 Value = 10 × 100 = 1000 After the split Shares = 20 Raw Price = 50 Value = 20 × 50 = 1000

The split changed the representation of the position, not its value. This is why a raw-price system must explicitly update shares when a split occurs.

Dividend example — $5 per share

Shares outstanding = the total number of shares that all investors together own Firm-level view before the dividend Shares Outstanding = 1,000 Raw Price = 100 Total Firm Value = 1,000 × 100 = 100,000 Dividend paid Dividend per Share = 5 Total Cash Leaving Firm = 1,000 × 5 = 5,000 Firm-level view after the dividend (assuming no market movement) New Firm Value = 95,000 New Raw Price ≈ 95 Your position before the dividend Shares = 10 Raw Price = 100 Your Position Value = 10 × 100 = 1000 Cash received Cash Received = 10 × 5 = 50 Your position after the dividend Shares = 10 Raw Price = 95 Stock Value = 10 × 95 = 950 Cash Received = 50 Total Investor Value = 950 + 50 = 1000

When a dividend is paid, value leaves the firm and goes to the shareholder. That is why the raw price typically drops by about the amount of the dividend on the ex-dividend date. If there is no market movement, the correct return is 0 once the cash received is included.

Spin-off example — value moves into a new asset

Shares outstanding = the total number of shares that all investors together own Firm-level view before the spin-off Shares Outstanding = 1,000 Raw Price = 100 Total Firm Value = 1,000 × 100 = 100,000 Spin-off event Suppose part of the firm worth 20,000 is spun off into a new company Firm-level view after the spin-off Original Firm Value = 80,000 New Spin-off Company Value = 20,000 Total Value = 80,000 + 20,000 = 100,000 Your position before the spin-off Shares = 10 Raw Price = 100 Your Position Value = 10 × 100 = 1000 Your position after the spin-off Original Stock Price = 80 Value of Your Original Shares = 10 × 80 = 800 Value of Your New Spin-off Shares = 200 Total Investor Value = 800 + 200 = 1000

In reality, a spin-off gives you shares in a new company, not cash. In this teaching dataset, selected spin-offs are simplified and treated like dividend-style cash events. That simplification is equivalent to assuming you immediately sell the new spin-off shares for cash. The value is the same; only the representation is simplified.

Key idea: splits change representation. Dividends and spin-offs move value. In every case, you want the system to keep track of where that value goes.

Keep it simple

You only need enough finance to understand what events change portfolio state.

Useful sentence

Adjusted price moves the price. A raw-price system moves the shares and cash.

Return reminder

A price drop is not always a loss. It may be a transfer of value.

Computational Thinking with Python — Part II2

Representation, uniform processing, and invariants

Why we use $$$$, why shares is a float, and what the invariant really means.

The possible confusion

You may reasonably ask: if stock shares should be integers, why store shares as a float? And if cash is a ticker, are the records really being processed the same way?

Two possible designs

DesignMain benefitMain cost
Cash separate from stock positionsEasy to say stock shares are integersPortfolio logic must special-case cash
Cash represented as ticker $$$$All records can be processed in one loopNeed an explicit invariant to keep stock-share counts valid

If cash is separate

cash = 1000.00 positions = [ {"ticker":"AAPL", "shares":10, "price":100}, {"ticker":"MSFT", "shares":5, "price":200} ] value = cash for pos in positions: value += pos["shares"] * pos["price"]

Here, cash is not processed the same way as stock positions. The code must handle cash separately and then loop through the stock positions.

If cash is a ticker

positions = [ {"ticker":"$$$$", "shares":1000.0, "price":1.0}, {"ticker":"AAPL", "shares":10.0, "price":100}, {"ticker":"MSFT", "shares":5.0, "price":200} ] value = 0.0 for pos in positions: value += pos["shares"] * pos["price"]

Now one loop works for every record. This is the main reason the representation is attractive.

So why the invariant?

Representation shares is stored as a float for every position-like record Invariant for all tickers except $$$$, shares == int(shares)

This is not a contradiction. It is a tradeoff. You simplified the processing code, so you must enforce a rule that keeps ordinary stock-share counts valid.

Key distinction: representation is about making computation easier. Invariants are about making the stored data valid.

Design choices that simplify the system

  • Reduce special-case logic
  • Give each file one clear job
  • Make important events explicit
  • Separate representation from invariants
  • Prefer local edits over unnecessary rewrites

A simpler design does not just make coding easier. It also makes debugging easier because there are fewer places where an error could be hiding.

Computational Thinking with Python — Part II4

Validation files built from the pricing data

Build files that help validate later user input.

Why create a market-date file?

A market-date file is useful because not every date that looks like a date is valid for portfolio actions. A date is valid only if it appears in the trading calendar represented by the pricing file.

Why a ticker list is not enough

A ticker is not simply valid forever. A ticker is valid during a specific interval of dates that matches the pricing file. So the universe file cannot just be a list of symbols. It must include start and end dates, and a ticker may need multiple records if it has multiple valid intervals.

One possible representation ticker, start_date, end_date Default convention end_date = 99999999 means valid through the end of the pricing table

AI prompt ideas

Prompt 1 I have a CSV pricing file with columns Date, Ticker, raw_close, adjusted_close. Write Python code that builds a sorted JSON file of unique market dates. Prompt 2 Using the same pricing file and the market-date file, write Python code that builds a ticker-universe file. For each ticker, find one or more continuous valid date ranges. Represent each range with start_date and end_date. Use 99999999 as the end_date when the ticker remains valid through the end of the pricing table.
Important follow-up: after AI writes the code, inspect the output visually. A program that runs is not automatically a program that is correct.

Computational Thinking with Python — Part II5

Using AI to build and test a data checker

AI can help build the first version, but understanding and testing remain essential.

How to use AI well here

A good use of AI is to upload the rebuilt files, explain what each file represents, and ask AI to create a data checker. Then ask AI a second question: what kinds of errors can this checker catch, and what kinds can it not catch?

That second question matters because a checker is always based on assumptions. Understanding those assumptions is part of understanding the data.

What the checker should try to catch

  • wrong columns
  • duplicate records
  • missing or impossible values
  • dividend or split dates not found in the pricing file
  • unexpected split records
  • split-adjusted price inconsistencies
  • ex-dividend drop mismatches
  • missing market-date rows for a ticker

How to test the checker properly

  1. Make a second copy of the data files. Never destroy the original data.
  2. Introduce deliberate errors into the copied files.
  3. Modify the checker so it reads the copied file names.
  4. Run the updated checker and see what it flags.

Why modify the code rather than ask AI to redo everything?

If only the file names changed, then the logic of the checker did not change. In that situation, it is usually better to make the local edit yourself than to ask AI to regenerate the whole program. Rewriting a working program can quietly change logic that was already correct.

Typical local edit PRICES_FILE = BASE_DIR / "prices_copy.csv" DIVIDENDS_FILE = BASE_DIR / "dividends_copy.csv" SPLITS_FILE = BASE_DIR / "splits_copy.csv"
Testing principle: a checker becomes meaningful only when you deliberately try to break the data and see what the checker notices.

Exercise 5 — Deliberately break the copied files

Create at least four errors in the copied data files. Try to include errors from different categories rather than four versions of the same mistake.

Good AI prompt pattern

Explain the files first, then ask for code, then ask for limitations, then inspect the result yourself.

What this process teaches

It deepens your understanding of both the files and the assumptions built into the checker.

Never do this

Do not overwrite original source data just because you want to test a checker.

Computational Thinking with Python — Part II6

The next design step — transactions as the event log

A simpler system has fewer files to consult later.

What the transaction file should be

The transaction file is the dated event log for the portfolio system. It should support at least these transaction types:

TYPEMeaning
BUYPurchase shares
SELLSell shares
CNTRBCash contribution
WDRWCash withdrawal
DIVDividend-style cash event added from the dividend file
SPLTSplit event added from the true-splits file

Why include DIV and SPLT here?

By including DIV and SPLT in the transaction file, later portfolio processing can rely on a single dated event log instead of repeatedly consulting separate event files. That reduces cognitive load.

At that point, the core system can be described using four main files:

  • ticker universe
  • pricing file
  • transactions file
  • market dates file

Each file has one clear purpose, which makes the system easier to design and easier to debug.

Should the transaction file include all dividend events?

Yes. The better design is to add all dividend records from the dividend file, not just the ones that match positions currently held in one account. Later transactions may make those records relevant, and the broader event history is useful for reconstruction and testing. The same idea supports adding all split events as SPLT.

What happens if you start from scratch?

If you choose “start from scratch,” the transaction file should still begin with the market events that already happened.

If you choose START FROM SCRATCH 1) Create a new transaction file 2) Load all DIV records into it 3) Load all SPLT records into it 4) Then allow manual transaction entry

Even if you do nothing, events still happen in the market. The transaction file records those events.

Key idea: the transaction file is broader than user-entered trades. It is the dated event log for the portfolio system.

Cognitive-load win

Putting DIV and SPLT into transactions means later code does not have to keep looking back into other event files.

Important distinction

Some transaction rows are system-generated from source data. Some are entered by a user.

Core files

Universe, prices, transactions, and market dates now form the main working system.

Computational Thinking with Python — Part II7

Wrapper, create_transaction(...), and seeded history

Keep session control separate from single-transaction logic.

Wrapper vs. single-transaction function

The one-time choice to start from scratch or load an existing transaction file belongs in a wrapper. That choice should be asked only once. After that, the program should stay in a loop so you can add multiple transactions until you exit.

Wrapper responsibilities - ask whether to start from scratch or load an existing file - create or load the transaction history - stay in a loop and repeatedly call create_transaction(...) create_transaction(...) responsibilities - create one valid transaction record - append it to the transaction structure/file - return control to the wrapper

Important invariants

Account-level invariant cash balance must never be negative Position-level invariant shares for any ticker must never be negative

Ask AI to generate the code

Prompt 3 — wrapper + create_transaction Write Python code for a portfolio transaction-entry system. Use a wrapper that asks only once whether to start from scratch or load an existing transaction file. If start from scratch is chosen, create a new transaction file and preload DIV and SPLT records. Then stay in a loop so multiple transactions can be added until exit. Create a function create_transaction(...) that adds one valid record at a time. Use uppercase TYPE values: BUY, SELL, CNTRB, WDRW, DIV, SPLT. Enforce these invariants: resulting cash balance must never be negative and resulting ticker share counts must never be negative.

Ask AI to generate an initial seeded transaction file

A second useful AI task is to create a plausible initial transaction file so the system can be tested without always starting from scratch. For this, upload your actual files so the seeded history is consistent with the real dates, tickers, prices, dividends, and splits.

Prompt 4 — seeded transactions from uploaded files Using the uploaded pricing file, market-date file, and ticker-universe file, create an initial CSV transaction file for this portfolio system with about 25 to 30 dated records. Use uppercase TYPE values: BUY, SELL, CNTRB, WDRW, DIV, SPLT. Use only valid market dates. Use only tickers valid on those dates. Use plausible prices based on the uploaded pricing file. Include multiple tickers, repeated buys and sells for the same ticker, several contributions and withdrawals, dividend events, and at least one split event. Keep the sequence plausible through time. Keep the resulting cash balance nonnegative and ticker share counts nonnegative.

What to do after AI generates the file

  1. Inspect the header row and check that the field names match the design.
  2. Scan the dates from top to bottom and make sure the sequence is plausible.
  3. Look for repeated tickers and verify that buys, sells, dividends, and splits all appear.
  4. Do a visual reasonableness check on cash activity and share activity.
  5. Then use the file to test portfolio reconstruction.
