Transactions Workbook: Student Version

In this lab, the goal is not just to write code. The goal is to think carefully about what should be stored, what should be reconstructed, what assumptions we are making, and how the design of our data affects the functions we can build.

This version is for discussion and discovery. Do not rush to code. Work through the questions and try to justify your design choices before looking at any polished summary.

1. Review of the pricing JSON files

Our market data is stored in two JSON files that contain the same underlying pricing facts in two different views.

View 1: prices_dates.json
Keyed by date.
View 2: prices_tickers.json
Keyed by ticker.

Each pricing record contains the following fields:

Field Meaning
date The trading date for the record.
ticker The stock symbol.
raw_price The actual closing price for that day.
dividend The dividend on that day.
shares_in The number of shares before a split.
shares_out The number of shares after a split.
Discussion prompt:
  • Why might it be useful to have both a date-keyed file and a ticker-keyed file?
  • What kinds of questions are easier to answer in each view?
  • What extra burden do we take on by keeping both?

10. User input design considerations

Good data design should lead to a reasonable user workflow.

Discussion prompt:
  • When the program starts, should it try to load an existing transaction file? Why?
  • If no transaction file exists, what should the program do?
  • Should the user have to type the date for every transaction?
  • Would it help to choose one working date and then enter several transactions for that date?
  • What state should persist across runs, and what state should exist only during one session?

11. The four functions in this lab

Function Purpose
create_transaction(...) Create one of four transaction types: contribution, withdrawal, buy, or sell.
get_cash_balance(as_of_date) Reconstruct the cash balance on a chosen day.
build_portfolio(as_of_date) Reconstruct the portfolio on a chosen day from transaction history.
list_transactions_for_ticker(ticker) Show the dated transaction history for one ticker.
