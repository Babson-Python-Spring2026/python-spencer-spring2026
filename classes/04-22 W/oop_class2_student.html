1. Reading Object State in JSON

Last class we learned how to build simple classes. A natural next question is: how might those objects be represented as plain data?

Example Structure
{
  "schools": [
    {
      "name": "Babson",
      "students": [
        {"name": "Alice", "student_id": 1},
        {"name": "Bob", "student_id": 2}
      ]
    },
    {
      "name": "MIT",
      "students": [
        {"name": "Eve", "student_id": 3}
      ]
    }
  ]
}

Important Observation
JSON can store state, but it cannot store methods. JSON tells us what the data is. It does not enforce whether the data is valid.

2. Classes That Use Other Classes

When one object contains other objects, the designer must decide what belongs where, who owns what data, and where important rules should be enforced.

Class Relationship
  • Student objects represent individual students.
  • School objects contain collections of students.
  • In larger systems, we often build objects out of other objects rather than keeping everything as loose dictionaries and lists.

3. Protecting Data with Properties

Definition
An invariant is a rule that should always remain true for an object.
Object Possible Invariant
Position shares should be a non-negative integer
CashAccount balance should stay non-negative (if overdrafts are not allowed)
School student IDs should be unique within one school
Class Discussion
  1. Which invariants are facts about the real world?
  2. Which invariants are design choices?
  3. Where should invariants be enforced?

One weak approach is to write special setter methods and hope people use them. A stronger approach is to control the normal path of access itself.

In-Class Exercise 3 — Protecting Shares in a Position Object

Goal: see why direct attribute access can break an object, and then redesign the class so shares are controlled.

Part 1 — Start with an Unprotected Class
class Position:
    def __init__(self, ticker, shares):
        self.ticker = ticker
        self.shares = shares

Run:

p = Position("AAPL", 10)
p.shares = -50
print(p.shares)

Questions:

  1. Did Python stop you?
  2. What rule seems to have been violated?
  3. Write the invariant in words.
Part 2 — Try a Weak Fix

Add a method like this:

def set_shares(self, shares):
    if shares < 0:
        raise ValueError("Shares must be non-negative")
    self.shares = shares

Then test:

p = Position("AAPL", 10)
p.set_shares(20)
p.shares = -5

Questions:

  1. Which line enforced the rule?
  2. Which line bypassed it?
  3. Why is this not enough?
Part 3 — Redesign with Protection

Rewrite the class so:

  • the real stored value is _shares
  • the public interface is shares
  • invalid assignments are rejected

Requirements:

  • __init__ should still accept shares
  • reading p.shares should work
  • assigning p.shares = value should validate
  • shares must be non-negative
Part 4 — Reflection
  1. What is the difference between _shares and shares?
  2. Which one is storage?
  3. Which one is part of the interface?
  4. Why is this design stronger?

Interface vs. Storage

In a property-based design, names like _shares and _balance are the stored values inside the object.

Names like shares and balance are part of the interface. They are the normal path by which outside code reads and updates the state.

One useful way to say this is:

  • Storage: _shares, _balance
  • Interface: shares, balance

4. Nice Object Behavior: __str__ and __add__

Objects are easier to work with if they can describe themselves clearly and define meaningful operations.

Example: __str__
class Student:
    def __init__(self, name, student_id):
        self.name = name
        self.student_id = student_id

    def __str__(self):
        return f"{self.name} ({self.student_id})"

Now print(student) produces a useful description.

Required AI Process
You may use AI, but you must use it as a design assistant, not as a machine that writes the whole solution for you.

Your submission must include three parts:

  1. Your Python solution
  2. A transcript or screenshots of your AI interaction
  3. A short reflection (about 1 page) answering the questions below
How to Use AI Correctly on This Assignment

Your AI interaction must show that you are designing the process. The goal is not to have AI produce a finished answer. The goal is to use AI to support decisions that you are making.

Before asking AI for code, you should already be thinking about:

  • what classes you need
  • what state each class should store
  • what transitions the system should support
  • what invariant must remain true

Good prompts often do things like:

  • ask AI to help you identify the state of each class
  • ask AI to suggest possible invariants
  • ask AI to critique a design that you already drafted
  • ask AI to help debug one small part after you have already written it
  • ask AI to compare two design choices that you are considering

Weak prompts give away the design work. Prompts like the ones below are a problem not only because they are too general, but because they hand control of the process to the AI:

  • “Write the whole program for me”
  • “Implement the complete solution”
  • “Give me the final answer”
  • “Do this for me”
  • “Write the code”

Your job is to design the process and stay in control of the code. AI should help you refine, test, or improve your design. It should not replace the design work itself.

Reflection Questions
  1. What was your invariant, and why did you choose it?
  2. What part of the design did you decide before asking AI anything?
  3. What question did you ask AI that was genuinely useful?
  4. What suggestion from AI did you reject or modify?
  5. How does your final design show that you controlled the process rather than delegating it?
Suggested Workflow
  1. Sketch the classes and state on paper or in comments.
  2. Write a first draft yourself.
  3. Ask AI for feedback on a small part of the design.
  4. Revise your code.
  5. Test your code with at least one invalid case.
  6. Submit code, AI transcript, and reflection together.
