Prompting as State, Transitions, Invariants

A computational-thinking view of prompting

Core idea

Prompting can be framed the same way you frame programming problems:

Prompting = building a system using State, Transitions, and Invariants.

This makes prompting feel less like “magic phrasing” and more like system design.

1. State — What exists right now?

State is what you tell the model before it starts working. You are defining the objects, the structure, and the representation.

Example

You are generating a transcript.

The input consists of:
- USER messages
- AGENT messages

Each message must be preserved exactly.
Bad
Make this into HTML
Better
The input is a sequence of USER and AGENT turns.
Each turn must remain unchanged.

2. Transitions — What is allowed to happen?

Transitions are the operations the model is allowed to perform. This is your algorithm.

Example

Perform the following steps:

1. Remove any line that begins with "Thought"
2. Preserve all other text exactly
3. Wrap each USER turn in <div class="user">
4. Wrap each AGENT turn in <div class="agent">
If you do not define transitions, the model invents them.

3. Invariants — What must never change?

Invariants are the guardrails. These are often the most important part of a strong prompt.

Example

Invariants:
- Do not change wording
- Do not reorder messages
- Do not summarize
- Do not combine turns

If an output “sounds nicer” but violates these rules, it is still wrong.

4. Output contract — What must the result look like?

This is similar to a type system. You specify the required form of the output.

Example

Output must be:
- a single valid HTML file
- with sections in this exact format:

<section class="user">
...
</section>

Putting it together

Here is a full STI-style prompt:

STATE:
The input is a sequence of messages labeled USER or AGENT.

TRANSITIONS:
1. Remove any line starting with "Thought"
2. Do not modify any other content
3. Wrap USER messages in <div class="user">
4. Wrap AGENT messages in <div class="agent">

INVARIANTS:
- Do not change wording
- Do not reorder messages
- Do not summarize
- Do not merge turns

OUTPUT:
Return a complete HTML file.

Useful advanced patterns

Pause control

Wait until I say "done" before processing

Verification step

First show me how you parsed the turns.
Do not generate final output yet.

Mode switching

You are now a strict parser, not a writer.

Error detection

If anything is ambiguous, stop and ask.

Idempotence

Running this twice should produce identical output.

What students should learn

  • AI is not helpful just because it is powerful. It becomes useful when the problem is represented clearly.
  • Vague prompts create vague systems.
  • Good prompting depends on explicit state, clear transitions, and strong invariants.
  • Prompting can be taught as a computational-thinking skill, not just as “tips and tricks.”
Better prompts come from better system design.
