🧠 PROMPTING CHEAT SHEET (STI FRAMEWORK)
1. 🟦 STATE — “What exists right now?”

This is what you tell the model before it starts working.

Examples
You are generating a transcript.

The input consists of:
- USER messages
- AGENT messages

Each message must be preserved exactly.

👉 You are defining:

the objects
the structure
the representation
Good vs Bad

❌ Bad (underspecified state)

Make this into HTML

✅ Good (explicit state)

The input is a sequence of USER and AGENT turns.
Each turn must remain unchanged.
2. 🟨 TRANSITIONS — “What is allowed to happen?”

These are the operations the model is allowed to perform.

Examples
Perform the following steps:

1. Remove any line that begins with "Thought"
2. Preserve all other text exactly
3. Wrap each USER turn in a <div class="user">
4. Wrap each AGENT turn in a <div class="agent">

👉 This is your algorithm

Key idea

If you don’t define transitions, the model invents them

That’s what happened earlier:

it summarized
it compressed
it “improved” text
3. 🟥 INVARIANTS — “What must NEVER change?”

This is the most important—and most underused.

Examples
Invariants:
- Do not change wording
- Do not reorder messages
- Do not summarize
- Do not combine turns

👉 These are your guardrails

Your transcript problem was an invariant failure

You wanted:

exact wording
exact order

But didn’t explicitly enforce it at first.

4. 🟩 OUTPUT CONTRACT — “What must the result look like?”

Think of this like a type system

Example
Output must be:
- a single valid HTML file
- with sections in this exact format:

<section class="user">
...
</section>
Why this matters

Without this, the model:

changes formatting
mixes styles
introduces inconsistencies
