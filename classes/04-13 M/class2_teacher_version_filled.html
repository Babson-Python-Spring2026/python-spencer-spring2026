Exercise 2.1 — What exactly happens if the same stock is bought twice?

Suppose the portfolio already contains AAPL and then buy_stock(port) is called and more AAPL is bought.

  1. Should the new purchase overwrite the old one?
  2. Should the function append a second AAPL entry?
  3. Should the two holdings be combined?
  4. What information might be lost under each choice?

Sample answer

The current code will append a second AAPL row. That means the portfolio may now contain two separate entries for the same ticker. That is not automatically wrong, but it forces us to decide what each row means.

  • If rows represent lots, then a second AAPL row may be acceptable because the purchases happened at different times and possibly at different prices.
  • If rows represent the current combined position, then the second purchase should probably be merged into the existing AAPL holding.
  • If the old row is overwritten, important history is lost.

The deeper issue is that the code has no clear model of whether it is storing positions, lots, or transactions. Until that is clarified, repeated buys are ambiguous.

Big idea: the issue is not just whether the code runs. The deeper issue is whether we have thought carefully enough about what state the function is changing, what repeated calls mean, and what the system is supposed to remember.

Thinking deeply before coding more

When we design or examine a function, we have to think deeply about exactly what we want to accomplish. We must think carefully about all the possibilities for how the state could change, especially when the function is called repeatedly. A shallow solution may look correct on the first call and still fail as a model of the system.

This leads to a broader lesson. More generally, understanding why things are done the way they are is critical to success in programming. Most recurring patterns are not accidents.

The same idea applies not just to the code we write, but to the language itself. When something appears repeatedly in Python, it is worth asking: is this a coincidence, or is there a real design reason behind it?

Consider the Python language. Look at the range function and slicing. What exactly is Python doing, and why?

Spanning the Possibility Space

Before coding, ask:
Have I considered all the ways this system can behave?

  • What states can exist?
  • What transitions are possible?
  • Where could things break?

Common failure modes:

  • Duplicate records
  • Missing data
  • Out-of-order rows
  • Invalid transitions
  • Non-trading days appearing in data

Key question:
What would have to go wrong for this system to fail?

Core idea

A representation can look fine at first and still break once the system evolves through time.

Broader point

Good programming involves understanding why representations are designed the way they are.

The inclusive-start, exclusive-end convention is easier to program with because it handles boundaries cleanly and gives the simple size rule end - start.

Inclusive start, exclusive end

At first, this may look like a small Python convention. But it is really an example of a deliberate representation choice.

Five reasons this pattern is so useful

ReasonWhy it helps
Length is simpleThe size is simply end - start, with no extra +1.
Concatenation is cleanAdjacent intervals fit together naturally without duplicate boundary values.
Gap checking is simpleTwo intervals meet cleanly when the first interval’s end equals the second interval’s start.
No gaps, no duplicatesEach value belongs to exactly one interval rather than being missed or counted twice.
Natural for state changesA new state can begin exactly at the boundary where the old one stops.

Exercise 2.3 — Apply the same idea to time

Suppose one interval runs from Jan 15 to Feb 10 and the next begins on Feb 10.

  1. What ambiguity appears if both intervals include Feb 10?
  2. How does inclusive start, exclusive end avoid that problem?

Sample answer

If both intervals include Feb 10, then Feb 10 belongs to both time ranges. That creates ambiguity: which state or price record should be used on that date?

With inclusive start and exclusive end, the first interval stops before Feb 10 and the next interval starts on Feb 10. So Feb 10 belongs to exactly one interval. That avoids double counting and makes the membership test simple:

start <= date < end
Key idea: good representations cover the possibilities exactly once. This Python pattern is a reminder that useful representations are chosen for reasons. The same standard should be used when modeling systems through time.

Pattern to notice

range and slicing follow the same rule. That is a clue that the rule is probably useful.

What to watch for

Compare what happens at the boundary value. Does the representation handle the boundary automatically, or does extra logic have to be added?

Looking ahead

This same idea will matter later when representing holdings through time.

Class 2 — Returning to the portfolio problem

Now apply the same standard of careful representation to the portfolio system.

The current model is still too shallow

The buy_stock function exposed one problem: repeated purchases of the same stock are not represented clearly. But the deeper problem is larger. The current design may let us store a portfolio at one moment, yet it does not let us answer many natural questions about the portfolio through time.

Exercise 2.4 — What questions should a real portfolio system answer?

Write down at least four questions that a useful portfolio system should be able to answer.

Examples of the kind of questions to think about:

Sample answer

Reasonable questions include:

  • What did I hold on a given date?
  • What was my cash balance on a given date?
  • What transactions happened for one ticker?
  • What was the portfolio worth on a given date?
  • How much has been contributed or withdrawn?
  • What gains have been realized and what gains are still unrealized?

The pattern is important: a real system must answer questions about both current state and history through time.

Important shift: once time matters, it is no longer enough to store only the current position. We need a way to record the changes that happened over time.

From positions to transactions

If the portfolio changes over time, one natural idea is to store the transactions rather than only the current holdings. A transaction is a dated event that changes the state of the portfolio.

Exercise 2.6 — Transactions as system history

If transactions are stored through time, what current state and historical information should the system be able to reconstruct from that history?

Sample answer

From transaction history, the system should be able to reconstruct:

  1. current holdings by ticker
  2. cash balance on any chosen date
  3. the dated history of one ticker or of the whole portfolio
  4. cost basis, realized gains/losses, and unrealized gains/losses

This is the key shift: transactions are the stored history, while portfolio state and reports are derived from that history.

Exercise 2.8 — What would the portfolio look like after a partial sale?

Suppose the portfolio begins with a contribution of $100,000. Then:

  1. Buy 500 shares of XYZ at $100 per share.
  2. Assume the price rises by $1 each day.
  3. After 10 days, sell 100 shares.

Assume no commissions and use FIFO. Build the portfolio report on the day of the sale. Show:

  • remaining shares
  • remaining cost basis
  • cash balance
  • market price
  • market value
  • realized gain/loss
  • unrealized gain/loss
  • total portfolio value

Extension: What should this report look like if all 500 shares were sold instead? Should XYZ still appear in the report?

Sample answer

Assume the buy happens at $100 and the sale occurs 10 days later at $110.

  • Initial contribution: $100,000
  • Buy 500 shares at $100 → cost $50,000
  • Cash after buy: $50,000
  • Sell 100 shares at $110 → proceeds $11,000
  • FIFO cost of shares sold: 100 × 100 = $10,000
  • Realized gain: $1,000

Remaining position:

  • Shares: 400
  • Remaining cost basis: $40,000
  • Market price: $110
  • Market value: 400 × 110 = $44,000
  • Unrealized gain: $44,000 - $40,000 = $4,000
  • Cash balance: $61,000
  • Total portfolio value: $61,000 + $44,000 = $105,000

If all 500 shares were sold instead, then XYZ should still appear in a historical report with zero shares so the realized gain remains visible.

Core move

Instead of storing only “what the portfolio is now,” store the dated events that caused the portfolio to become what it is now.

Notice the connection

The portfolio is a dynamic system. Once time matters, the representation must account for every state change through time.

Representation reminder

A static snapshot may hide the very history needed to answer the most important questions.

Class 2 — From transactions to valuation through time

Once transactions are stored, the next natural question is what the portfolio is worth on any date.

What else should a portfolio system do?

If transactions let us reconstruct the holdings, then a natural next question is:

What is the portfolio worth on any given date?

That question introduces a new requirement. It is not enough to know how many shares are held. We also need a way to find the relevant prices on the date being examined.

Exercise 2.10 — What is needed to value the portfolio through time?

List the kinds of information the system would need in order to compute portfolio value on any trading day.

Sample answer

To value the portfolio through time, the system needs at least:

  • transaction history
  • a valid market date file
  • a pricing table keyed by ticker and date
  • ticker availability information
  • rules or data for dividends, splits, and other corporate actions when relevant

In short, one file tells us what we hold, and the market data ecosystem tells us what it is worth.

Data resources the system will need

To value the portfolio through time, the system will need more than a transactions file. It will need a small data ecosystem.

Design note: one practical choice is to store dates as YYYYMMDD. That representation sorts naturally and can sometimes simplify early programs because many comparisons can be done directly without importing date objects right away.

Exercise 2.11 — Build a ticker availability file

Create a sp100_tickers.json file that defines when each ticker is tradable using half-open intervals [start, end).

Use these conventions:

  • dates must follow the market date convention YYYYMMDD
  • intervals are stored as [start, end)
  • if a ticker is still active, use 99999999 as the end date

Suppose ticker XYZ trades on every market date in January, does not trade in February, and then trades again from March onward. How should that be represented?

{ "XYZ": [ {"start": "20250101", "end": "20250201"}, {"start": "20250301", "end": "99999999"} ] }

Why is this better than assuming every ticker has only one start date and one end date?

Sample answer

The interval representation is useful because a ticker may disappear and later reappear. One start date and one end date would incorrectly assume continuous existence.

Using [start, end) intervals:

  • membership is easy to test with start <= date < end
  • adjacent intervals fit together cleanly
  • gaps in availability can be represented directly

This is exactly the same design advantage we saw with range and slicing.

A pricing table through time

A pricing table is often stored one row per date. But for this course, it may be useful to think about a representation that also makes the valid interval explicit. For example, a pricing row could include a start date and an end date for the period during which the row is valid.

Every point in time should map to exactly one valid representation of the portfolio and exactly one relevant price record.

Class 2 — Corporate actions and data interpretation

A real system must decide what different corporate actions mean for state, not just for returns.

Not all data fields mean what they first appear to mean

Once prices, dividends, and corporate actions are introduced, another important modeling question appears:

What does the data mean for the state of the portfolio?

A split, a dividend, and a spin-off may all affect value, but they do not affect the state of the portfolio in the same way.

EventWhat changes in the portfolio system?
DividendCash is distributed; share count usually stays the same
SplitShare count changes; price adjusts; value is preserved
Spin-offA new asset appears; value leaves one asset and appears in another

Exercise 2.15 — Why is a spin-off not just a split?

Write a short explanation of why a spin-off should not be modeled the same way as a true stock split.

Sample answer

A split changes the share count of the same asset. A spin-off is different because a new asset appears. Value leaves one security and appears in another. So a spin-off changes not just the number of shares but the actual set of assets being held. That means it should be modeled as a change in portfolio state, not merely as a scaling of one existing position.

Using AI to help inspect the data and build reports

Teacher note

This section is a good place to emphasize that AI should be used after students understand the files. The real skill is not “can AI write code?” but “can the student define the right relationship to test and recognize when AI made a hidden assumption?”

One useful role for AI is to help write small programs that test ideas about the data or build reports from transaction history. For example, if close prices are price-only and adjusted close prices reflect total return, a short program can help check whether that relationship seems to hold and whether any rows look anomalous. In the same spirit, AI can help draft functions that reconstruct cash balance, portfolio state, or ticker history from transaction records.

But before asking AI to write that code, you really have to get your hands dirty and carefully examine the structure and content of each input file. What are the columns called? What does each field mean? What is one row supposed to represent? Are dates unique within a ticker? Are duplicate rows possible? Are there missing rows? Do dividends, splits, and prices appear to be aligned the way you think they are?

That matters because if the problem is not understood, AI may write code that looks plausible and is still wrong. It may check the wrong relationship, use the wrong key, or quietly assume a data structure that does not actually exist. Understanding the data and understanding the code remain essential.

A very good way to test whether your checking code really works is to deliberately introduce errors into otherwise clean files and see whether the program catches them. You can duplicate rows, delete rows, or change values. That kind of testing is often more informative than simply running the code once on data that appears fine.

AI reminder: AI can help test an idea, but only if the idea is understood clearly enough to ask the right question, inspect the output, and recognize when the code is based on the wrong assumptions. Good prompts begin with careful examination of the actual files.

Closing idea for this section: the problem is often not that the data is “bad.” The problem is that data may be designed for one purpose and then used for a different purpose. A return-oriented representation is not always the right representation for a state-based portfolio system.

Representation matters again

Yahoo-style adjusted data is useful for return analysis. A state-based portfolio system needs raw events and explicit changes in holdings and cash.

System question

Does this event change cash, shares, or the set of assets being held?
