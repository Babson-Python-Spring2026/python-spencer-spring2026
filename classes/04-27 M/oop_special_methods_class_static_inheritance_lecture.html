Opening Mental Model: What Is an Object?

Concept An object combines state and behavior. State is the data the object remembers. Behavior is what the object can do. 
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

account = BankAccount("Luna", 1250.50)
Mental model account is a reference to one object. That object has state: owner and balance. Methods are functions that operate on that object through self.

STI framing

STI Part Meaning in OOP
State Attributes such as self.balance, self.owner, self.name
Transitions Methods that change or use state, such as deposit(), withdraw(), speak()
Invariants Rules that should remain true, such as balance should not become negative unless overdrafts are allowed

Part 1: Special Methods — __str__ and __add__

Concept Special methods let our objects work naturally with Python syntax. __str__ controls printing. __add__ controls what + means.

1.1 __str__: How should this object print?

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    def __str__(self):
        return f"{self.owner}'s balance is ${self.balance:,.2f}"

account = BankAccount("Luna", 1250.5)
print(account)
Expected output: 
Luna's balance is $1,250.50
Warning __str__ must return a string. It should usually not print directly.

1.2 __add__: What should + mean?

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    def __str__(self):
        return f"{self.owner}'s balance is ${self.balance:,.2f}"

    def __add__(self, other):
        return self.balance + other.balance

a1 = BankAccount("Luna", 1250.50)
a2 = BankAccount("Rex", 300.25)

print(a1 + a2)
Mental model a1 + a2 becomes a1.__add__(a2). Python passes a1 as self and a2 as other.

Design choice: should __add__ return a number or an object?

Return a number

def __add__(self, other):
    return self.balance + other.balance

Good when + means “calculate a total.”

Return a new object

def __add__(self, other):
    return BankAccount(
        "Combined",
        self.balance + other.balance
    )

Good when + means “combine into a new object.”

Part 2: Instance Methods, Class Methods, and Static Methods

Concept Python methods differ by what they receive automatically: self, cls, or neither.
Method type First parameter Best use
Instance method self Use or change one object’s state
Class method cls Use or change class-level data, or create objects in alternate ways
Static method None automatically Utility function logically connected to the class

2.1 Instance method

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    def deposit(self, amount):
        self.balance += amount

deposit() needs self because it changes one specific account.

2.2 Class method

class BankAccount:
    bank_name = "Babson Bank"

    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    @classmethod
    def change_bank_name(cls, new_name):
        cls.bank_name = new_name

change_bank_name() works at the class level. It changes data shared by the class.

2.3 Class method as an alternate constructor

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    @classmethod
    def open_empty_account(cls, owner):
        return cls(owner, 0)
Mental model cls means “the class being used.” This is often better than hard-coding BankAccount(...), especially when inheritance is introduced later.

2.4 Static method

class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner
        self.balance = balance

    @staticmethod
    def is_valid_amount(amount):
        return amount >= 0

is_valid_amount() does not need one object and does not need the class. It is attached to the class because it belongs conceptually with bank account logic.

Part 3: Inheritance

Concept Inheritance represents an is-a relationship. A Cat is an Animale. A Dog is an Animale. 
class Animale:
    def __init__(self, name):
        self.name = name

    def __str__(self):
        return f"My name is {self.name}"

    def speak(self):
        return "The mighty animale roars"


class Cat(Animale):
    def speak(self):
        return "meow meow meow"


class Dog(Animale):
    def speak(self):
        return "Bow wow"

3.1 One object, initialized in layers

class Animale:
    def __init__(self, name):
        self.name = name
        print("Animale init")


class Cat(Animale):
    def __init__(self, name):
        print("Cat init starts")
        super().__init__(name)
        print("Cat init ends")


c = Cat("Luna")
Output: 
Cat init starts
Animale init
Cat init ends
Mental model Cat("Luna") creates one object. super().__init__(name) does not create another object. It initializes the parent part of the same object.

3.2 Overriding

If a child class defines a method with the same name as the parent, the child method wins.

c = Cat("Luna")
d = Dog("Rover")

print(c.speak())   # Cat version
print(d.speak())   # Dog version

3.3 Calling the parent method explicitly

c = Cat("Luna")

print(c.speak())            # Cat.speak(c)
print(Animale.speak(c))     # force Animale.speak(c)
Concept An instance method can be called through the class if you manually pass the object: Animale.speak(c). Here, c is supplied as self.

3.4 Polymorphism

animals = [Cat("Luna"), Dog("Rover")]

for animal in animals:
    print(animal.speak())
Mental model The list is “just a list of objects,” but the design idea is stronger: it is a list of objects that share the same interface. Each object responds to speak() in its own way.

3.5 Multiple inheritance and MRO

class A:
    def speak(self):
        return "A"

class B:
    def speak(self):
        return "B"

class C(A, B):
    pass

print(C().speak())   # A
print(C.__mro__)
Warning In simple inheritance, super() usually feels like “call the parent.” More precisely, super() calls the next method in the MRO.
