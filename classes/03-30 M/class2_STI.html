Thinking deeply before coding more

When we design or examine a function, we have to think deeply about exactly what we want to accomplish. We must think carefully about all the possibilities for how the state could change, especially when the function is called repeatedly. A shallow solution may look correct on the first call and still fail as a model of the system.

This leads to a broader lesson. More generally, understanding why things are done the way they are is critical to success in programming. Most recurring patterns are not accidents.

The same idea applies not just to the code we write, but to the language itself. When something appears repeatedly in Python, it is worth asking: is this a coincidence, or is there a real design reason behind it?

Consider the Python language. Look at the range function and slicing. What exactly is Python doing, and why?

Spanning the Possibility Space

Before coding, ask:
Have I considered all the ways this system can behave?

  • What states can exist?
  • What transitions are possible?
  • Where could things break?

Common failure modes:

  • Duplicate records
  • Missing data
  • Out-of-order rows
  • Invalid transitions
  • Non-trading days appearing in data

Key question:
What would have to go wrong for this system to fail?

Core idea

A representation can look fine at first and still break once the system evolves through time.

Broader point

Good programming involves understanding why representations are designed the way they are.

Inclusive start, exclusive end

At first, this may look like a small Python convention. But it is really an example of a deliberate representation choice.

Five reasons this pattern is so useful

ReasonWhy it helps
Length is simpleThe size is simply end - start, with no extra +1.
Concatenation is cleanAdjacent intervals fit together naturally without duplicate boundary values.
Gap checking is simpleTwo intervals meet cleanly when the first interval’s end equals the second interval’s start.
No gaps, no duplicatesEach value belongs to exactly one interval rather than being missed or counted twice.
Natural for state changesA new state can begin exactly at the boundary where the old one stops.

If the portfolio changes over time, one natural idea is to store the transactions rather than only the current holdings. A transaction is a dated event that changes the state of the portfolio.

Core move

Instead of storing only “what the portfolio is now,” store the dated events that caused the portfolio to become what it is now.

Notice the connection

The portfolio is a dynamic system. Once time matters, the representation must account for every state change through time.

Representation reminder

A static snapshot may hide the very history needed to answer the most important questions.

Naming convention reminder

When field names begin to multiply, aim for names that are easy to understand, short, and used consistently across similar fields.

For example, a name like trn_type makes the role of the field explicit. Short coded values such as CNTRB and WDRW may also be useful if they are documented clearly and used consistently.

Once transactions are stored, the next natural question is what the portfolio is worth on any date.

What else should a portfolio system do?

If transactions let us reconstruct the holdings, then a natural next question is:

What is the portfolio worth on any given date?

That question introduces a new requirement. It is not enough to know how many shares are held. We also need a way to find the relevant prices on the date being examined.

Class 2 — Corporate actions and data interpretation

A real system must decide what different corporate actions mean for state, not just for returns.

Not all data fields mean what they first appear to mean

Once prices, dividends, and corporate actions are introduced, another important modeling question appears:

What does the data mean for the state of the portfolio?

A split, a dividend, and a spin-off may all affect value, but they do not affect the state of the portfolio in the same way.

EventWhat changes in the portfolio system?
DividendCash is distributed; share count usually stays the same
SplitShare count changes; price adjusts; value is preserved
Spin-offA new asset appears; value leaves one asset and appears in another

Using AI to help inspect the data and build reports

One useful role for AI is to help write small programs that test ideas about the data or build reports from transaction history. For example, if close prices are price-only and adjusted close prices reflect total return, a short program can help check whether that relationship seems to hold and whether any rows look anomalous. In the same spirit, AI can help draft functions that reconstruct cash balance, portfolio state, or ticker history from transaction records.

But before asking AI to write that code, you really have to get your hands dirty and carefully examine the structure and content of each input file. What are the columns called? What does each field mean? What is one row supposed to represent? Are dates unique within a ticker? Are duplicate rows possible? Are there missing rows? Do dividends, splits, and prices appear to be aligned the way you think they are?

That matters because if the problem is not understood, AI may write code that looks plausible and is still wrong. It may check the wrong relationship, use the wrong key, or quietly assume a data structure that does not actually exist. Understanding the data and understanding the code remain essential.

A very good way to test whether your checking code really works is to deliberately introduce errors into otherwise clean files and see whether the program catches them. You can duplicate rows, delete rows, or change values. That kind of testing is often more informative than simply running the code once on data that appears fine.

AI reminder: AI can help test an idea, but only if the idea is understood clearly enough to ask the right question, inspect the output, and recognize when the code is based on the wrong assumptions. Good prompts begin with careful examination of the actual files.

Closing idea for this section: the problem is often not that the data is “bad.” The problem is that data may be designed for one purpose and then used for a different purpose. A return-oriented representation is not always the right representation for a state-based portfolio system.

Representation matters again

Yahoo-style adjusted data is useful for return analysis. A state-based portfolio system needs raw events and explicit changes in holdings and cash.

System question

Does this event change cash, shares, or the set of assets being held?
